# Superpowers Wrapper Design

## Context

`superpowers@openai-curated` is installed from the OpenAI curated plugin snapshot. The user wants an independent local wrapper project at `/Users/j7an/Documents/github/superpowers-wrapper` because `/Users/j7an/Documents/github/superpowers` will be removed in the future.

The wrapper should install as `superpowers@superpowers-wrapper` and source runtime plugin files from `https://github.com/obra/superpowers`. Codex requires plugin manifests to include a `version` field, so the local wrapper cannot omit a version entirely. Instead, the wrapper generates a ref-aware wrapper version with a build fingerprint and records the actual upstream Superpowers identity separately.

## Goals

- Create a local Codex marketplace wrapper for upstream Superpowers.
- Keep upstream Superpowers as the source of truth.
- Default to tracking the latest stable upstream release tag.
- Allow explicit refs such as `v6.0.3`, prerelease tags, or commit SHAs when requested.
- Make it easy to probe whether the installed wrapper is current.
- Avoid direct edits to Codex internals, curated plugin cache, or `~/.codex/config.toml`.
- Avoid automated removal of existing or conflicting plugins.

## Non-Goals

- Do not publish a public marketplace.
- Do not vendor upstream Superpowers history into this repository.
- Do not automatically uninstall `superpowers@openai-curated`.
- Do not add a wrapper flag that removes curated Superpowers.
- Do not remove any plugin other than the wrapper's own installed `superpowers@superpowers-wrapper` entry when refreshing the wrapper install requires it.
- Do not use the GitHub Releases API.
- Do not treat the wrapper manifest version as the upstream Superpowers version.

## Operator Responsibility

The wrapper does not manage plugin name collisions. Before relying on wrapper-provided `superpowers:*` skills, the operator is responsible for removing or disabling any existing same-name or similar plugin provider, including `superpowers@openai-curated`.

The wrapper may print a note about this responsibility, but it must not run removal commands or provide a removal flag.

## Architecture

The project is a local Codex marketplace root. It contains committed wrapper metadata and scripts. Runtime Superpowers files are generated by copying selected files from an upstream checkout into `plugins/superpowers`.

The generated plugin tree is a normal local Codex plugin source. This avoids relying on symlink behavior during Codex plugin installation and keeps the installed package self-contained.

## Marketplace

The committed marketplace file exposes one plugin:

- Marketplace name: `superpowers-wrapper`
- Plugin name: `superpowers`
- Installed plugin id: `superpowers@superpowers-wrapper`
- Source path: `./plugins/superpowers`
- Installation policy: `AVAILABLE`
- Authentication policy: `ON_INSTALL`
- Product policy: `["CODEX"]`
- Category: `Developer Tools`

Keeping the plugin name as `superpowers` preserves skill names such as `superpowers:brainstorming`.

## Upstream Ref Resolution

`config/upstream-ref` contains the default desired upstream ref. The default value is:

```text
latest-release
```

`latest-release` is a wrapper keyword, not a Git ref. It means:

1. Query upstream tags with Git.
2. Keep stable semver tags matching `vMAJOR.MINOR.PATCH`.
3. Ignore prerelease tags unless they are passed explicitly.
4. Sort by numeric semantic version, not lexical string order.
5. Select the highest stable version.
6. Resolve the selected tag to the peeled commit SHA, not the annotated tag object SHA.

The wrapper must use Git tag data only, for example `git ls-remote --tags https://github.com/obra/superpowers`. It must not call the GitHub Releases API.

When `git ls-remote --tags` returns both `refs/tags/vX.Y.Z` and `refs/tags/vX.Y.Z^{}`, the wrapper must use the `^{}` peeled line as the release commit. If the implementation fetches the tag locally instead, it must use an equivalent commit-peeling command such as `git rev-list -n1 <tag>`.

Explicit refs override `config/upstream-ref`. Examples:

```text
v6.0.3
v6.1.0-beta.1
896224c4b1879920ab573417e68fd51d2ccc9072
```

Explicit prerelease tags are allowed because the user intentionally requested them.

## Versioning

Codex requires `.codex-plugin/plugin.json` to contain a valid version. The
wrapper therefore derives a version from the resolved upstream ref rather than
from the wrapper repository itself.

- Stable tags such as `v6.0.3` keep their SemVer base as `6.0.3`.
- Prerelease tags such as `v6.1.0-beta.1` keep their SemVer base as
  `6.1.0-beta.1`.
- `main` uses a SemVer base of `0.0.0-main`.
- Other named refs use a SemVer base of `0.0.0-ref-<sanitized-ref>`.
- Raw 40-character commit SHAs use `0.0.0+wrapper.<short-sha>`, where
  `short-sha` is the first seven characters of the resolved upstream commit.
- Stable tags, prerelease tags, `main`, and other named refs also carry
  `+wrapper.<short-sha>` so humans can trace generated artifacts back to the exact
  resolved upstream commit.

This is the wrapper package version, not the upstream Superpowers release version. It changes when the prepared upstream commit changes and gives humans a visible fingerprint in the manifest, but update detection must not rely on semver precedence for build metadata. The authoritative update comparison is the upstream commit recorded in `.superpowers-upstream.json`.

If Codex does not refresh an already installed local plugin with `codex plugin add`, the wrapper refresh flow may remove and re-add only the wrapper's own `superpowers@superpowers-wrapper` installation. It must not remove curated Superpowers or any other conflicting provider.

## Committed Manifest Requirements

The committed wrapper manifest at `plugins/superpowers/.codex-plugin/plugin.json` must satisfy the local Codex plugin validator without relying on generated upstream metadata. It must include:

- top-level `name`, `version`, and `description`
- `author.name`
- `skills: "./skills/"`
- `interface.displayName`
- `interface.shortDescription`
- `interface.longDescription`
- `interface.developerName`
- `interface.category`
- `interface.capabilities`
- `interface.defaultPrompt` or `interface.default_prompt`

The manifest may include optional URLs, keywords, brand color, and icon paths only when those values validate. It must omit unsupported top-level fields such as `hooks`, even when upstream includes them.

If the manifest includes `interface.composerIcon` or `interface.logo`, `scripts/prepare` must copy the referenced assets before plugin validation runs. The implementation plan should either keep icons and enforce this ordering, or omit icon fields to avoid coupling validation to generated assets.

## Upstream Metadata

`scripts/prepare` writes generated metadata to:

```text
plugins/superpowers/.superpowers-upstream.json
```

The file records the exact upstream source used for the generated plugin tree:

```json
{
  "source": "https://github.com/obra/superpowers",
  "requested_ref": "latest-release",
  "resolved_ref": "v6.0.3",
  "commit": "896224c4b1879920ab573417e68fd51d2ccc9072",
  "upstream_manifest_version": "6.0.3"
}
```

`requested_ref` records what the operator or config asked for. `resolved_ref` records the tag or ref selected after resolution. `commit` is the authoritative comparison value.

## Commands

### `scripts/probe`

Resolves the desired upstream ref, reads generated and installed metadata when available, and reports whether the wrapper is current.

Expected output includes:

- requested ref
- resolved ref
- desired commit
- generated plugin commit, if present
- installed wrapper commit, if detectable
- status: current, needs prepare, needs install, or unknown

`scripts/probe` must not modify files.

### `scripts/prepare`

Fetches or updates `.cache/upstream/superpowers`, resolves the desired ref, copies upstream runtime files into `plugins/superpowers`, writes `.superpowers-upstream.json`, and updates the wrapper manifest fingerprint.

`scripts/prepare` modifies only the wrapper repository.

### `scripts/install`

Registers or refreshes the local marketplace as needed and installs or refreshes `superpowers@superpowers-wrapper` through normal Codex CLI commands.

The wrapper command maps to current Codex CLI verbs:

- `codex plugin marketplace add /Users/j7an/Documents/github/superpowers-wrapper` for marketplace registration, and for marketplace refresh if empirical testing shows local marketplaces are snapshotted
- `codex plugin add superpowers@superpowers-wrapper` for install
- `codex plugin remove superpowers@superpowers-wrapper` only if empirical testing shows that remove-then-add is required to refresh the wrapper's own installed cache

`scripts/install` must not remove other plugins. It should print a short note that collision management is the operator's responsibility.

### `scripts/update`

Runs the normal update flow:

1. Probe desired upstream state.
2. Prepare if generated plugin files are stale.
3. Install if the installed wrapper is stale or missing.

`scripts/update` must not remove other plugins.

## Generated Plugin Contents

The generated plugin should copy the upstream files needed for Codex runtime behavior:

- `skills/`
- `assets/`
- `hooks/` when present, even if the wrapper manifest omits unsupported `hooks` fields
- `LICENSE`
- `README.md`
- `CODE_OF_CONDUCT.md`

The wrapper manifest remains committed and controlled by the wrapper project. It should not be overwritten wholesale by upstream `.codex-plugin/plugin.json`, because local Codex validation may reject upstream fields such as `hooks`.

Copying upstream `hooks/` preserves upstream runtime files, but hook activation is not assumed. Because the local validator rejects a top-level `hooks` field today, the implementation plan must include an empirical check for whether Codex activates `hooks/session-start-codex` without a manifest hook declaration. If it does not, the wrapper still provides skills, but automatic session-start activation is out of scope until Codex supports a validated hook declaration.

## Validation

The wrapper should validate both its own scripts and the generated plugin:

- Shell tests for ref resolution, metadata writing, and version fingerprinting.
- A fake local upstream repository test to avoid network dependence in unit tests.
- Codex plugin validation with `/Users/j7an/.codex/skills/.system/plugin-creator/scripts/validate_plugin.py plugins/superpowers` after prepare.

Manual validation includes:

- `scripts/probe`
- `scripts/prepare`
- plugin validator
- `scripts/install`
- `codex plugin list --json`

The implementation plan must answer these empirical questions before relying on install behavior:

1. For a local marketplace, does `codex plugin add` read the current source tree, or does `codex plugin marketplace add` create a snapshot that must be refreshed?
2. Does refreshing an installed local plugin require `codex plugin remove superpowers@superpowers-wrapper` followed by `codex plugin add superpowers@superpowers-wrapper`?
3. Where does Codex cache the installed wrapper plugin, and can `scripts/probe` read `.superpowers-upstream.json` from that cache?
4. Does Codex activate copied upstream session-start hooks without a validated manifest `hooks` field?

## Error Handling

Commands fail with clear messages when:

- Git is unavailable.
- The upstream repo cannot be reached.
- No stable semver tag exists for `latest-release`.
- An explicit ref cannot be resolved.
- Required upstream paths are missing.
- Plugin validation fails.
- Codex CLI commands fail during install.

`scripts/prepare` should stage generated files in a temporary directory before replacing the existing generated plugin contents. A failed prepare should not destroy the previous working generated tree.

## Security And Maintenance

The wrapper executes only local shell scripts and Git/Codex commands. It does not depend on GitHub API auth, package managers, or new runtime dependencies.

Keeping uninstall and collision handling outside the wrapper reduces coupling to Codex CLI behavior and avoids brittle maintenance if Codex changes remove commands or plugin naming semantics.
